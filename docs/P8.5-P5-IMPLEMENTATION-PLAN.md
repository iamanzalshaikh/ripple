## 0. Read this first

### What P8.5-P5 is

> Make Ripple capable of understanding the desktop environment and **safely operating across any application** using the same pipeline:

```text
Voice → NLU → L0/L2 Planner → Validator → Permission Engine → Tool Executor → Tools → Observe/Recover
```

### What P8.5-P5 is NOT

| Wrong | Right |
|-------|-------|
| Cursor integration phase | Cursor is one **target app** |
| WhatsApp/Gmail adapter phase | Adapters stay; P5 adds **generic** layers |
| “Add 100 tools” | **33 reliable tools**, frozen |
| New planner architecture | Same brain — stronger **hands** |
| `developer.*` namespace | All dev workflow tools live under **`automation.*`** |
| CDP/Puppeteer as default browser layer | **Extension + desktop input**; CDP dev-only (`RIPPLE_USE_CDP=1`) |

### Frozen categories (from `toolTypes.ts`)

```text
desktop | filesystem | browser | system | memory | communication | automation | ai
```

No 9th category. No `developer.*`, `app.*`, `input.*`, `window.*`, `vision.*`.

### Two frozen implementation rules (P5)

1. **`automation.open_project`** — IDE-agnostic. Entity resolver → project detect → `desktop.launch_app`. No Cursor/VS Code hardcode. `preferred_ide` is **P6 memory**.
2. **`ai.generate_action_plan`** — draft only. Output re-enters `planValidator` → permission → executor. **Never** GPT → execute.

## 2. Safety model (universal — required before P5 ships)

Every **state-mutating** P5 tool uses the same executor chain:

```text
Planner produces ExecutionPlan
        ↓
planValidator (args, paths, batch caps)
        ↓
Risk classification (low | medium | high on tool definition)
        ↓
permissionPass1 (category + utterance policy)
        ↓
permissionPass2 (resolved paths, wildcards, system paths)
        ↓
confirmStepIfNeeded (overlay dialog)
        ↓
pushUndoBeforeMutate (snapshot where possible)
        ↓
tool.execute()
        ↓
observe + verify result
        ↓
recoveryEngine on failure
```

### Risk tiers (frozen)

| Tier | P5 tools | Behavior |
|------|----------|----------|
| **low** | `list_directory`, `search`, `read_file`, `get_metadata`, `get_active_window`, `extract_text`, `find_element`, `find_code`, `summarize_screen`, `extract_context` | Execute immediately |
| **medium** | `create_file`, `write_file`, `move_file`, `launch_app`, `close_app`, `type_text`, browser interact | Rate limit; confirm on destructive |
| **high** | `patch_file`, `run_command`, `run_script`, `git_operation`, `run_tests` | **Always confirm** + undo where feasible |

### Files to extend

| File | Change |
|------|--------|
| `toolExecutorSafety.ts` | Add `FILESYSTEM_MUTATORS`: `write_file`, `patch_file`, `create_file`, `move_file`; add `AUTOMATION_MUTATORS` set; wire confirm kinds |
| `toolTypes.ts` | Document risk defaults per category |
| `permissionEngine.ts` | Add `automation` + `ai` policy rules |
| `executionGuard.ts` | New confirm kinds: `patch_file`, `run_command`, `run_script`, `git_push`, `run_tests` |
| `undoStack.ts` | File backup before `patch_file` / `write_file` |
| `planValidator.ts` | Path traversal guard, max read bytes, command injection patterns for `automation.*` |
| `toolExecutor.ts` | Post-step verify hooks for automation + filesystem |

### `automation.open_project` — IDE-agnostic (frozen)

**Rule:** P5 must **never** hardcode Cursor, VS Code, or any IDE name inside `automation.open_project`.

The tool resolves a **project folder**, detects project signals, then launches whatever IDE the system resolves — not a brand.

```text
automation.open_project
        ↓
entityResolver (folder name / path / "Horizon backend")
        ↓
filesystem.search + list_directory (confirm project root)
        ↓
detect project markers: package.json, .git, src/, pyproject.toml, etc.
        ↓
resolve IDE app name (generic registry lookup — NOT "Cursor" string)
        ↓
desktop.launch_app(resolvedIdeApp)
```

**P5 default (no user memory yet):** use OS/default IDE discovery (`nativeAppRegistry`, installed apps scan) — e.g. first known IDE on machine, or open folder in Explorer if no IDE found.

**P6 (later):** `preferred_ide = Cursor` (or VS Code, IntelliJ, …) comes from **planner memory / user preferences** — injected into resolver, still not hardcoded in the tool.

**Example — user says:** *"Open Horizon backend"*

```text
filesystem.search("Horizon")
        ↓
find folder C:\Projects\Horizon-backend
        ↓
detect: package.json + .git + src/
        ↓
automation.open_project(path)
        ↓
desktop.launch_app(detected IDE)    ← resolver picks app; P5 has no Cursor branch
```

**Forbidden in P5:**

```text
if (ide === "Cursor") launchCursor()   ❌
openProjectInCursor(path)              ❌
```

### AI tools — draft only, never execute (frozen)

**Rule:** No AI tool may call `tool.execute()` on mutating tools or bypass the pipeline.

`ai.generate_action_plan` (and `ai.reason_about_task`) produce **structured output only**. Execution always re-enters the normal path:

```text
ai.generate_action_plan
        ↓
ExecutionPlan (draft)
        ↓
planValidator
        ↓
Permission Engine (pass 1 + pass 2)
        ↓
confirmStepIfNeeded (if mutating)
        ↓
Tool Executor
        ↓
observe / recover
```

**Forbidden:**

```text
GPT decides → execute directly          ❌
ai.generate_action_plan → runCommand    ❌
AI tool calls filesystem.patch_file     ❌
```

Implementation: `ai.generate_action_plan` returns `{ plan: ExecutionPlan }` as tool **output**; the planner or orchestrator must explicitly pass that plan through `validatePlan` + executor — same as L0/GPT planner output today.

Other `ai.*` tools (`summarize_screen`, `extract_context`, `detect_element`) are **read-only context** — safe to run inside executor, but their output still cannot trigger side effects.

1. Unit: execute handler with mocked OS/**extension bridge**/GPT (CDP only in dev-path tests)
